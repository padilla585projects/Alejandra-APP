================================================================================
  ROLES Y ACCESO
================================================================================

- superadmin  → Código ADMIN_CODE. Ve todo, elige departamento al entrar.
- encargado   → Código de obra. Gestiona su departamento asignado.
- operario    → Código de obra. Solo lectura/scan en su departamento.

Departamentos:
  ⚡ Eléctrico  → Bobinas + PEMP + Carretillas
  🔧 Mecánicas  → Solo PEMP + Carretillas (sin Bobinas)
  🔺 Seguridad  → Solo escáner/scan (busca matrícula cross-dept)

  -F "metadata={\"main_module\":\"worker.js\",\"compatibility_date\":\"2024-01-01\",

  Sin esto, env.DB queda undefined y toda la app deja de funcionar.

================================================================================
  DECISIONES TÉCNICAS IMPORTANTES
================================================================================

  - ESTADO_APP.txt se actualiza automáticamente después de cada cambio,
    sin necesidad de que el usuario lo pida
  - version.json: archivo en cada repo que se actualiza con cada versión.
    La app lo lee al arrancar (cache:'no-store') y si la versión difiere,
    limpia caches y recarga sola. ACTUALIZAR SIEMPRE junto con sw.js/index.html.
  - empresa_admin: rol de admin de empresa propia. Tiene acceso total a sus
    datos (obras, usuarios, inventario, sesiones, ideas) pero no a config global
  - Método de trabajo: EDITS QUIRÚRGICOS sobre archivos existentes,
    NUNCA reescritura completa
  - sendBeacon debe usar Blob({type:'application/json'}) — si se pasa
    string plano, el Content-Type es text/plain y el worker no lo parsea
  - batchUpdate de Sheets: separar add y delete en dos peticiones distintas
  - Nombres de pestañas Sheets: solo ASCII (sin emojis) para evitar
    problemas de encoding en la URL
  - SW: mecanismo completo de auto-actualización (v3.4):
    · skipWaiting + clients.claim + postMessage SW_ACTUALIZADO
    · La página escucha message + controllerchange + visibilitychange
    · Si hay SW en waiting al cargar → SKIP_WAITING inmediato
    · Reload con timestamp para evitar HTTP cache
  - Seguridad: nunca guardar ADMIN_CODE en el código, siempre en
    Cloudflare Secrets
  - Auth v3.9: sistema de tokens en D1. getAuth() prueba X-Token primero,
    luego fallback a headers legacy. crearSesion() inserta en tabla sesiones.
    cerrarSesionServidor() borra el token. Frontend guarda token (no código).
