================================================================================
  QUÉ ES EL AGENTE ALEJANDRA
================================================================================

Alejandra es un agente IA (Claude Sonnet vía Anthropic API, con Haiku de router
y GPT-4o/GPT-4o-mini como apoyo) integrado en la app Alejandra APP. Sigue
teniendo DOS workers Cloudflare distintos (esto no ha cambiado, verificado hoy
que ambos worker.js siguen existiendo):

1. LA APP PRINCIPAL — worker.js (raíz del repo), wrangler.toml (raíz)
   - name/URL: "alejandra-app-api" →
     https://alejandra-app-api.alejandra-app.workers.dev
   - Deploy: .github/workflows/deploy-worker.yml (paths: worker.js,
     wrangler.toml — el de la raíz, NO alejandra-agente/wrangler.toml)
   - Alejandra vive aquí desde las versiones antiguas vía devAIChat()/
     handleDevAI() (worker.js raíz, líneas ~2638 y ~12810) — canal Telegram y
     endpoint /dev/ai-chat. Chat accesible desde la app móvil (index.html).
   - Este archivo (ALEJANDRA_AGENTE.txt) NO cubre este worker.js en detalle —
     su enfoque es alejandra-agente/worker.js (punto 2). Si tocas devAIChat/
     handleDevAI, audita ese archivo aparte antes de asumir que las reglas de
     aquí aplican igual.

2. EL AGENTE INDEPENDIENTE — alejandra-agente/worker.js (con lógica NEXUS
   avanzada, L0-L4, tools, gating de auth) — es el foco de TODO el resto de
   este documento.
   URL:   https://alejandra-agente.alejandra-app.workers.dev
   Panel: https://padilla585projects.github.io/Alejandra-APP/alejandra-panel.html
          (también panel.html, servido por GitHub Pages)
   Auth panel: ver sección "AUTENTICACIÓN" más abajo

IMPORTANTE: ambos workers comparten el MISMO D1 (alejandra-db, mismo
database_id en los dos wrangler.toml) y el mismo R2 (alejandra-app-files) —
son la misma Alejandra, una sola base de datos multi-empresa (aislada por
empresa_id), servida por dos Workers Cloudflare separados con su propio
wrangler.toml y su propio pipeline de deploy.

================================================================================
  ARQUITECTURA DEL AGENTE (alejandra-agente/worker.js)
================================================================================

NEXUS ROUTER:
  1. Haiku (MODEL_ROUTER = claude-haiku-4-5) clasifica el mensaje → determina
     qué experto de NEXUS_EXPERTS atiende (simple/app/tecnico/web/reflexion/
     completo/ingenieria).
  2. Se construye el system prompt de ese experto combinando sus módulos de
     NEXUS_MODULES (ver "PROMPT EN CAPAS L0-L4" abajo).
  3. Se llama a Sonnet (MODEL_EXPERTO = claude-sonnet-4-6) con las tools de
     TOOLS_POR_EXPERTO filtradas además por auth (ver "GATING DE TOOLS").
  4. Bucle de iteraciones si hay tool calls, hasta un máximo.

NEXUS_MODULES (fragmentos de prompt, 20 módulos a fecha de hoy):
  Para la lista completa y actualizada de nombres de módulo (ajusta el rango
  de líneas si NEXUS_MODULES/NEXUS_EXPERTS se han movido):
    grep -n "^  [a-z_]*:" alejandra-agente/worker.js | awk -F: '$1<897'
  (o simplemente abrir worker.js y mirar el objeto NEXUS_MODULES, línea ~31).
  Incluye, entre otros: base, app, tecnica, nexus, evolucion, web, reflexion,
  decision, contexto_sesion, aprendizaje_proactivo, ram, formato, razonamiento,
  ingenieria, ingenieria_electrica, capacidades_avanzadas, inteligencia_negocio,
  asistente_escaneo, seguimiento_proactivo, proactividad_real.

NEXUS_EXPERTS (perfiles, definidos en worker.js línea ~897):
  simple     → módulos mínimos, respuestas rápidas/cortas
  app        → uso cotidiano de la app (bobinas, fichajes, PRL, obras...)
  tecnico    → preguntas sobre la infraestructura/código del propio sistema
  web        → búsqueda web (usa GPT-4o-mini + web_search_preview de OpenAI)
  reflexion  → auto-mejora, memoria, decisiones autónomas
  completo   → combina casi todos los módulos, para consultas amplias
  ingenieria → cálculos eléctricos/mecánicos de obra (cable, bandeja, protección,
               esquemas)
  Todos usan MODEL_EXPERTO salvo el router (MODEL_ROUTER para clasificar).
  maxTokens y la lista exacta de módulos por experto: ver worker.js línea ~897
  (NEXUS_EXPERTS) — varían con frecuencia, no se transcriben aquí.

PROMPT EN CAPAS L0-L4 (buildAnthropicSystemBlocks, worker.js línea ~916):
  L0 — Identidad estática (módulos 'base' + 'formato') → cacheada (prompt
       caching de Anthropic, TTL 5 min), igual para todos los expertos.
  L1 — Módulos propios del experto activo → cacheados junto con L0.
  L2 — Contexto dinámico: pending thoughts, reglas destiladas, self-knowledge.
  L3 — Estado live del sistema (salud, usuarios activos, etc.).
  L4 — Catálogo de tools visibles para ESTE experto (tras el gating de auth).
  Motivo de la separación: L0/L1 no cambian entre turnos de la misma sesión ni
  entre usuarios → se benefician del prompt caching de Anthropic (ahorro real
  de coste/latencia). L2-L4 sí cambian por turno/usuario y van fuera del bloque
  cacheado.

HERRAMIENTAS (TOOLS_POR_EXPERTO, worker.js línea ~1867):
  A fecha de hoy hay 56 tools distintas definidas, repartidas de forma no
  uniforme entre los 7 expertos (el experto "app", por ejemplo, tiene acceso a
  la mayoría; "web" solo a 3; "simple" a 3 básicas). Lista completa y ordenada:
    grep -n "name: '" alejandra-agente/worker.js | grep -oP "name: '\K[a-z_0-9]+(?=')" | sort -u
  Categorías representativas (no exhaustivo): memoria (memory_save/read,
  ram_save/read/clear), archivos (listar/ver/subir_archivo, files R2), BD
  (consultar_bd/escribir_bd — ver gating IDOR abajo), código/deploy
  (github_leer/escribir/buscar, grep_codigo, patch_codigo, ejecutar_deploy,
  verificar_deploy, rollback, test_endpoint, nexus_manage), ingeniería
  (calcular_cable/bandeja/proteccion, generar_esquema_electrico,
  listar/borrar_esquema, analizar_foto_obra), gestión de obra (estado_obra,
  gestionar_tarea/rfi/oc/acta/calidad), comunicación (enviar_email,
  enviar_push, enviar_telegram_informe, iniciar_conversacion), planificación
  (pensar, planificar, descubrir_herramientas, recuperar_conversacion),

================================================================================
  GATING DE TOOLS POR AUTENTICACIÓN Y AISLAMIENTO MULTI-EMPRESA
  (fixes de seguridad de la sesión 04-05/07/2026 — leer esto antes de tocar
  auth, tools de BD, archivos, o el endpoint de test)
================================================================================

Historial del problema: hasta la sesión del 04/07/2026, /api/chat y
/api/chat/stream confiaban ciegamente en el `usuario_id` que mandaba el CLIENTE
en el body (sin verificar token), y esa identidad decidía qué tools podía usar
Claude. Cualquiera podía mandar {usuario_id:"adrian"} sin token y desbloquear
patch_codigo/github_escribir/ejecutar_deploy/rollback (escritura de código +
deploy) o consultar_bd/escribir_bd (lectura/escritura arbitraria de la BD
compartida de TODAS las empresas). Se corrigió con 3 capas, todas en vigor:

1. filtrarToolsPorAuth() (lib.js) — gating por identidad VERIFICADA:
     authOk          = true SOLO si la identidad viene de un Authorization:
                        Bearer <token> verificado contra la tabla `sesiones` o
                        ADMIN_TOKEN (ver getAuth() en worker.js) — NUNCA del
                        usuario_id que manda el body.
     esDevVerificado = true SOLO si esa identidad verificada es developer/admin.
   TOOLS_SOLO_DEV_VERIFICADO (patch_codigo, github_escribir, ejecutar_deploy,
   rollback, test_endpoint, configurar_alerta) exige esDevVerificado.
   TOOLS_REQUIEREN_SESION (consultar_bd, escribir_bd, listar_archivos,
   ver_archivo, exportar_datos) exige authOk.

   configurar_alerta se añadió a la lista de solo-dev el 05/07/2026
   (continuación 14): la acción "crear" guardaba un condicion_sql arbitrario
   que "verificar" ejecutaba tal cual más tarde sin ninguna validación —
   cualquier sesión podía persistir un UPDATE/DELETE/DROP disfrazado de
   "alerta". exportar_datos se añadió a solo-sesión el mismo día: exportaba
   bobinas/personal/fichajes/materiales/gastos de TODAS las empresas sin
   filtrar por empresa_id, y concatenaba obra_id/fecha_desde/fecha_hasta como
   string sin parametrizar (inyección SQL directa). Ver capa 2 (más abajo)
   para el resto del fix de exportar_datos.

2. validarScopeEmpresaBD() (lib.js) — aislamiento por empresa_id para
   consultar_bd/escribir_bd: el gating de (1) exige sesión, pero no evita que
   un usuario de la empresa A consulte/escriba datos de la empresa B (la query
   SQL es texto libre generado por el modelo). Esta capa solo permite tablas de
   negocio conocidas con empresa_id real en el esquema de producción
   (TABLAS_EMPRESA_PERMITIDAS, 58 tablas a fecha de hoy — lista completa en
   lib.js línea ~44) y exige que la query filtre explícitamente por el
   empresa_id REAL del que llama (derivado de la sesión, nunca del texto de la
   query). Se excluyen a propósito `sesiones` y `vincular_tokens` aunque tengan
   empresa_id: contienen tokens de sesión/vinculación, y leerlos permitiría
   suplantar a otro usuario de la misma empresa (incluido un admin). También
   bloquea columnas sensibles (password_hash). esDevVerificado se salta esta
   capa. El modo "custom" de exportar_datos (sql_custom) reutiliza esta misma
   función desde el 05/07/2026 (continuación 14), en vez de solo comprobar que
   empezara por "SELECT" como antes.

   validarSoloSelectBD() (lib.js) — comprobación "solo SELECT, sin verbos de
   escritura" extraída de consultar_bd para compartirla también con
   configurar_alerta (condicion_sql, validado al crear Y en cada ejecución de
   "verificar" como defensa en profundidad) y con el modo custom de
   exportar_datos (sql_custom). Las 5 exportaciones fijas de exportar_datos
   (bobinas/personal/fichajes/materiales/gastos) además pasaron de construir
   el WHERE por concatenación de string a placeholders + bind() (helper
   construirWhere en worker.js), con empresa_id obligatorio salvo dev
   verificado.

3. empresaDeArchivo() / puedeAccederArchivo() (worker.js, junto a
   /files/<key>) — aislamiento por empresa_id para archivos servidos desde R2:
   antes, /files/<key> y las tools listar_archivos/ver_archivo servían
   CUALQUIER key de R2 a CUALQUIER sesión autenticada, sin comprobar que el
   archivo pertenece a su empresa. Los objetos no llevan empresa_id en el key
   (se generó en /upload, sin sesión verificada) — solo
   `customMetadata.usuario_id`. Se resuelve ese usuario_id contra la tabla
   `usuarios` para saber la empresa dueña; si no se puede determinar con
   certeza → se trata como NO accesible (fallar cerrado, no abierto).

Anti-SSRF en test_endpoint: urlPermitidaTestEndpoint() (lib.js) — allowlist de
host (solo *.alejandra-app.workers.dev sobre https, con comprobación real de
subdominio, no substring) para evitar que la tool test_endpoint se use para
sondear red interna o servicios arbitrarios.

Rate limiting y tope de gasto diario (protección anti runaway-cost): antes
/api/chat y /api/chat/stream no tenían NINGUNA protección de volumen ni de
gasto. Ahora (worker.js, validarRateLimit/validarTopeGastoDiario):
  - 15 peticiones/minuto por identidad (usuario_id si hay sesión, si no la IP),
    vía RATE_LIMIT_KV, ventana deslizante de 1 minuto.
  - Tope global de 10 $/día (todavía no hay empresa_id en
    alejandra_token_uso, así que el tope es global, no por empresa — separar
    por empresa queda como mejora futura).
  - Fail-open ante error de KV/D1 (no bloquear el servicio si el propio
    rate limiter falla).

Todas las funciones puras de este bloque (filtrarToolsPorAuth,
extraerTablasQuery, validarScopeEmpresaBD, urlPermitidaTestEndpoint,
esStatusReintentableAnthropic, calcularEsperaReintentoMs,
calcularCosteYProveedor) viven en alejandra-agente/lib.js — ver siguiente
sección.

================================================================================
  TESTS (alejandra-agente/lib.js + lib.test.js) — desde commit 93d0d3a
================================================================================

worker.js es un único archivo de ~9.000 líneas sin router ni framework de test.
Para poder testear al menos la lógica de más riesgo sin reescribir todo el
worker, se extrajeron las funciones PURAS (sin D1/KV/fetch reales) a un módulo
sibling `lib.js`, que worker.js importa. Solo vive ahí lo que NO toca env.DB,
env.RATE_LIMIT_KV, env.FILES ni hace fetch() real:

  - PRECIOS_USD + calcularCosteYProveedor  — pricing/coste por modelo+proveedor
  - filtrarToolsPorAuth (+ TOOLS_SOLO_DEV_VERIFICADO/TOOLS_REQUIEREN_SESION)
  - extraerTablasQuery / validarScopeEmpresaBD (+ TABLAS_EMPRESA_PERMITIDAS/
    COLUMNA_BLOQUEADA_BD)                    — fix IDOR de consultar_bd/escribir_bd
  - validarSoloSelectBD                      — "solo SELECT", compartida entre
    consultar_bd, configurar_alerta y exportar_datos (fix continuación 14)
  - urlPermitidaTestEndpoint (+ HOSTS_PERMITIDOS_TEST_ENDPOINT) — anti-SSRF
  - esStatusReintentableAnthropic / calcularEsperaReintentoMs — backoff/retry
    de fetchAnthropicConReintentos

Todo lo que necesita I/O real (registrarTokenUso, validarRateLimit, getAuth,
empresaDeArchivo, etc.) se queda en worker.js, sin cobertura automatizada
todavía (verificación manual vía curl tras cada deploy). Ampliar esto con

decide invertir en tests de integración de rutas reales.

Cómo correr los tests:
  cd alejandra-agente && npm ci && npm test        (vitest run, 47 tests)

El workflow de deploy (deploy-alejandra-agente.yml) instala dependencias y
corre `npm test` ANTES de aplicar migraciones D1 y desplegar — si los tests
fallan, el deploy no continúa (ver sección "DEPLOY Y CI/CD").

================================================================================
  AUTENTICACIÓN
================================================================================

El panel usa tokens almacenados en alejandra_tokens (D1). getAuth() (worker.js)
comprueba, para las rutas que la usan:
  1. Authorization: Bearer <token> contra la tabla `sesiones` (identidad real
     de usuario logueado en la app/panel) — es la fuente de verdad para authOk/
     esDevVerificado (ver sección de gating).
  2. Para el panel admin específicamente: verificarAdminToken(env, token, req)
     -- si env.ADMIN_TOKEN existe y coincide → autorizado (env var como
     respaldo); si no, consulta alejandra_tokens WHERE token=? AND tipo='admin'
     AND activo=1 AND (expires_at IS NULL OR expires_at > datetime('now')).

TOKEN ACTUAL: el configurado por wrangler secret ADMIN_TOKEN (ver "SECRETS" en
la sección de deploy) o el que haya configurado el usuario en el panel — NO
asumir un valor fijo, puede haberse rotado.

FIX 05/07/2026 (continuación 14, hallazgos #2 y #3 del mismo audit que
configurar_alerta/exportar_datos — código incluido, sin querer, en el commit
ce6e7d9 de otro agente trabajando en paralelo sobre el mismo repo, cuyo
mensaje solo habla de "pagina Planos IA en panel.html"; ver SESION.md
continuación 14/15 para la aclaración completa):
  - Hallazgo #3: verificarAdminToken() no tenía NINGÚN límite de intentos
    propio -- se podía probar tokens repetidamente contra /admin/migrate,
    /api/admin/*, /push, /api/reflexion, /conocimiento sin throttling. Ahora
    recibe `req` (opcional) y aplica el mismo validarRateLimit() de /api/chat
    por IP, bucket propio "admin-auth:ip:...", fail-open si KV falla.
  - Relacionado: la columna `expires_at` de alejandra_tokens existe desde el
    schema original pero JAMÁS se comprobaba en la query de verificación --
    un token con fecha de expiración pasada seguía siendo válido para
    siempre. Se añadió la comprobación `(expires_at IS NULL OR expires_at >
    datetime('now'))`.
  - Hallazgo #2: POST /auth/verify-session devolvía env.ADMIN_TOKEN en texto
    plano en la respuesta a cualquier sesión superadmin/desarrollador válida
    -- si esa respuesta se filtraba (XSS, log, sesión interceptada), el
    atacante se quedaba con el secreto maestro, estático y sin expirar, más
    duradero que la propia sesión robada. Ahora emite un token efímero nuevo
    en alejandra_tokens (tipo='admin', expira a las 12h) en vez del secreto
    -- el panel (alejandra-panel.html) no necesita cambios porque ya trataba
    "token" como un bearer opaco guardado en localStorage.
  - Limitación conocida, no bloqueante: los tokens efímeros no se limpian
    automáticamente al expirar (solo dejan de ser válidos vía la comprobación
    de expires_at) -- se van acumulando como filas inactivas-por-fecha en
    alejandra_tokens. Limpieza periódica queda como mejora futura si el
    volumen llega a ser un problema.

HISTORIA DEL BUG (por qué hay dos mecanismos de admin):
  - migrate_001 usaba INDEX inline en CREATE TABLE (inválido en SQLite/D1) →
    fallaba silenciosamente → alejandra_tokens vacía.
  - migrate_003 lo corrige: crea tablas con sintaxis correcta + INSERT OR
    REPLACE del token.
  - env.ADMIN_TOKEN (wrangler secret) se setea en el workflow como respaldo.

GITHUB ACTIONS: .github/workflows/deploy-alejandra-agente.yml
  Trigger: push a main en paths alejandra-agente/** o el propio workflow (o
  workflow_dispatch manual).
  Orden de pasos:
    1. Checkout + setup Node 22 + instalar wrangler.
    2. cd alejandra-agente && npm ci   (instala vitest, ver "TESTS" arriba)
    3. cd alejandra-agente && npm test (vitest run) — SI FALLA, el deploy NO
       continúa (ver siguiente paso, viene después en el workflow).
    4. Aplicar migrate_001/002/003 (con || echo para no fallar si ya existen).
    5. npx wrangler deploy.
    6. Setear secrets: ANTHROPIC_API_KEY, OPENAI_API_KEY, ADMIN_TOKEN (solo en
       push, no en workflow_dispatch).
    7. Health check final (GET /health, espera HTTP 200 tras sleep 20s).

  .github/workflows/pages.yml — Trigger: push a main. Copia index.html,
  panel.html, alejandra-panel.html, sw.js, etc. a _site y despliega a
  GitHub Pages.

WRANGLER.TOML (alejandra-agente/wrangler.toml):
  name: "alejandra-agente" · main: worker.js
  [[d1_databases]]   DB              → alejandra-db (0c9eccde-78f1-476d-ac68-bf452bec0c62)
  [[r2_buckets]]     FILES           → alejandra-app-files

                     añaden más).

================================================================================
  REGLAS PARA MODIFICAR EL AGENTE
================================================================================

1. El agente está en alejandra-agente/worker.js (NO en el worker.js raíz, si
   existe uno para la app principal — confirmar cuál es cuál antes de editar).
2. Las funciones PURAS (sin D1/KV/fetch) van en lib.js, no en worker.js — y
   si las tocas, actualiza/añade tests en lib.test.js (ver "TESTS" arriba).
   El workflow de deploy falla el build si los tests no pasan: úsalo a tu
   favor, no lo veas como un obstáculo.
3. Versión propia del agente, desacoplada de la PWA (fix commit a27d650):
   el comentario de cabecera de worker.js decía "v6.03" mientras /health
   devolvía "6.12" — desincronizados. Causa raíz: ninguno de los dos era un
   contador propio del agente, ambos se "prestaron" en su día del changelog
   de la PWA (ESTADO_APP.txt), que lleva su propia numeración totalmente
   independiente (la PWA va ya por v7.55, sin relación con el agente).
   Corregido: ahora el agente tiene su propio contador (v6.13 en adelante).
   Regla a partir de ahora: sube el número en la cabecera Y en la respuesta
   de GET /health A LA VEZ, solo cuando cambie ESTE worker (alejandra-agente/
   worker.js), y NUNCA lo copies de ESTADO_APP.txt/version.json de la PWA —
   son changelogs de proyectos distintos aunque compartan D1/R2.
4. Si cambias tablas D1 → crear migrate_00X_*.sql y añadirlo al workflow.
5. Si tocas consultar_bd/escribir_bd o cualquier cosa de aislamiento por
   empresa_id → lee primero la sección "GATING DE TOOLS..." de este archivo,
   y añade/ajusta tests en lib.test.js. Este es el área con más historial de
   bugs de seguridad (IDOR) del sistema — máxima cautela.
6. El agente usa ES modules (import/export, `export default { fetch() }`).
7. No usar el `wrangler.toml` del raíz para el agente — usar
   alejandra-agente/wrangler.toml.
8. Para desplegar solo el agente manualmente:
     cd alejandra-agente && npm ci && npm test && npx wrangler deploy
9. Para ver logs en tiempo real:
     npx wrangler tail --name alejandra-agente
10. Nunca commitear secretos (API keys, tokens) en ningún archivo — solo vía
    GitHub Actions secrets / `wrangler secret put`.
